# Lock-Free Очередь на основе CAS с использованием Hazard Pointers

Данный проект представляет собой реализацию неблокирующей (lock-free) очереди на базе алгоритма Майкла и Скотта (Michael & Scott) с безопасным управлением памятью через Hazard Pointers. Проект включает в себя встроенный тестовый стенд для проверки корректности (инвариантов) и измерения производительности в сравнении со стандартной блокирующей очередью (`std::queue` + `std::mutex`).

## Использование и запуск

Программа принимает 6 аргументов командной строки. 
**Формат запуска:**
```bash
./LockFree_Queue <prod_count> <cons_count> <ops_per_thread> <load_ops> <load_threads> <load_push_ratio>
```
*   `prod_count`, `cons_count`, `ops_per_thread` — параметры для функционального теста (проверка корректности).
*   `load_ops` — количество операций на один поток в нагрузочном тесте.
*   `load_threads` — количество потоков в нагрузочном тесте.
*   `load_push_ratio` — процент операций `enqueue` (остальное — `dequeue`). Значение от 1 до 99.

## Сценарии тестирования

В соответствии с заданием, тестовый стенд реализует два основных направления тестирования.

### 1. Тест на корректное завершение и отсутствие гонок данных
Данный тест запускается первым при любом вызове программы. Он создает заданное число производителей и потребителей, которые генерируют уникальные сообщения. По завершении потоков проверяется отсутствие потерь, отсутствие дубликатов и сохранение порядка FIFO.

**Команда запуска (пример):**
```bash
./LockFree_Queue 4 4 1000 100000 2 50
```

**Ожидаемый результат:**
Программа успешно отрабатывает без зависаний (deadlocks). Деструкторы корректно освобождают память (retire_lists очищаются), после чего выводится сообщение:
```text
--- Starting functial testing (Producers: 4 | Consumers: 4) ---
Functional test succeded. Invariants were saved!
```

### 2. Сценарии нагрузочного тестирования (Производительность)

Ниже приведены сценарии запуска для различных профилей нагрузки и ожидаемые результаты, полученные опытным путем на тестовом стенде.

#### Сценарий А: Низкая и средняя конкуренция (Оптимальный режим)
Количество потоков не превышает количество логических ядер процессора. Нагрузка сбалансирована (50% Push / 50% Pop).

**Команды запуска:**
```bash
./LockFree_Queue 4 4 1000 2000000 4 50
./LockFree_Queue 4 4 1000 1000000 8 50
```
**Ожидаемое время и результат:**
*   **Время:** ~0.75 - 0.85 сек (LF) против ~0.83 - 0.90 сек (Blocking).
*   **Пропускная способность:** Lock-Free очередь стабильно обходит блокирующую (~10.6 млн оп/сек против ~9.5 млн оп/сек на 4 потоках).
*   **Задержка (Latency):** Средняя задержка у Lock-Free ниже (~150 нс против ~180 нс). Пиковые задержки практически идентичны.

#### Сценарий Б: Экстремальная конкуренция (Оверсубскрипция потоков)
Множество потоков (до 100) одновременно конкурируют за доступ к очереди, вызывая сильное вытеснение планировщиком ОС.

**Команда запуска:**
```bash
./LockFree_Queue 4 4 1000 100000 100 50
```
**Ожидаемое время и результат:**
*   **Время:** ~1.3 - 1.4 сек (LF) против ~1.6 - 1.7 сек (Blocking).
*   **Пропускная способность:** Lock-Free очередь превосходит блокирующую (~7.5 млн оп/сек против ~5.8 млн оп/сек).
*   **Задержка (Latency):** Средняя задержка у Lock-Free в 2 раза лучше (~3500 нс против ~8500 нс). 
*   **Специфика (Max Latency):** Из-за эффекта "громкой толпы" (thundering herd) и постоянных провалов CAS-циклов у неудачливых потоков, максимальная задержка (max latency) у Lock-Free может достигать 0.8 - 1.1 секунды, в то время как у блокирующей очереди мьютекс сглаживает максимальную задержку до ~3-8 мс.

#### Сценарий В: Асимметричная нагрузка (Влияние узких мест)
Тестирование поведения при смещении нагрузки в сторону добавления (Push-heavy) или извлечения (Pop-heavy) на 32 потоках.

**Команда запуска (75% Push / 25% Pop):**
```bash
./LockFree_Queue 4 4 1000 250000 32 75
```
**Команда запуска (25% Push / 75% Pop):**
```bash
./LockFree_Queue 4 4 1000 250000 32 25
```
**Ожидаемое время и результат:**
*   **Преобладание Push (75%):** Lock-Free очередь может *уступать* блокирующей (~5.5 млн оп/сек против ~6.0 млн оп/сек). Это связано с высокой конкуренцией 32 потоков за единственный указатель `tail`, что приводит к огромному количеству повторов цикла (retries) и инвалидации кэш-линий.
*   **Преобладание Pop (25%):** Lock-Free очередь показывает *выдающиеся* результаты, уничтожая блокирующую реализацию (~9.9 млн оп/сек против ~6.6 млн оп/сек). Операция `dequeue` в реализованном алгоритме масштабируется значительно лучше.
